# 시연 시나리오 — AI 홈페이지 파인더

광주광역시 북구청을 대상으로 한 AI 홈페이지 파인더 데모 시나리오입니다.

## 사전 준비

```bash
cd 400-ai-finder
PYTHONPATH=. .venv/bin/python scripts/run_all_demos.py \
    --site-id bukgu_gwangju \
    --snapshot tests/fixtures/bukgu_gwangju_demo_snapshot.json
```

*`--provider`를 생략하면 기본 DeepSeek 프리셋(`deepseek-primary`)이 자동 적용됩니다.
Snapshot 모드이므로 네트워크/API 키 없이 안정적으로 동작합니다.*

실행 후 두 화면을 각각 열어둡니다:

- **왼쪽 모니터(또는 탭)**: 모바일 사용자 화면 — http://localhost:8400
- **오른쪽 모니터(또는 탭)**: 운영자 대시보드 — http://localhost:8090

---

## 시연 흐름 (약 7분)

### 1. 인사 및 배경 설명 (30초)

> "공공기관 홈페이지는 메뉴가 많고 복잡해서 시민들이 원하는 정보를 찾기 어렵습니다.
> AI 홈페이지 파인더는 자연어로 질문하면 해당 메뉴와 문서를 찾아 안내하는 서비스입니다.
> 오늘은 광주광역시 북구청을 대상으로 시연하겠습니다."

---

### 2. 모바일 사용자 화면 시연 (3분)

**화면**: http://localhost:8400

#### 2-1. 첫 화면 소개 (20초)

> "일반 시민이 보는 화면입니다. ChatGPT 스타일의 채팅 인터페이스로 구성되어 있어서, 누구나 익숙하게 사용할 수 있습니다."

화면 확인:
- 상단: 사이트명 (광주광역시 북구청) + 다크모드 토글 버튼
- 중앙: 대화 영역 (환영 메시지)
- 하단: 입력창 + 추천 질문 Chip (민원서식, 교육접수, 정보공개, 고시공고)

> "아래에 자주 찾는 질문 버튼이 있어서 바로 시작할 수 있습니다."

#### 2-2. 민원서식 질문 (30초)

**[민원서식]** Chip 클릭

> "시민이 '민원서식 어디서 받아?'라고 물어보면, AI가 답변을 생성합니다."

화면 확인:
- 사용자 메시지: "민원서식 어디서 받아?"
- AI 답변: 마크다운 형식의 안내 (답변 → 관련 자료 → 다음 단계 → 확인 필요)
- 답변 하단: 관련 홈페이지 카드 구조 (제목 + URL + 유형)

> "답변과 함께 관련 홈페이지 링크를 바로 제공해서 시민이 클릭하면 해당 페이지로 이동합니다.
> 하단에 '정확한 최신 내용은 연결된 공식 홈페이지에서 확인해 주세요'라는 안내가 포함됩니다."

#### 2-3. 교육접수 질문 (30초)

**[교육접수]** Chip 클릭

> "이번에는 '교육접수는 어디서 해?'라고 물어보겠습니다."

화면 확인:
- 메시지가 누적되어 이전 대화와 함께 표시
- 출처 1건, 교육접수 메뉴 링크

#### 2-4. 자유 질문 (30초)

입력창에 **"고시공고는 어디서 확인해?"** 입력 후 **Enter** 또는 **[전송]** 클릭

> "시민이 직접 질문을 입력할 수도 있습니다. 고시공고 관련 메뉴를 찾아줍니다."

#### 2-5. 다크모드 토글 (15초)

다크모드 버튼 클릭

> "다크모드도 지원합니다. 야간이나 저조도 환경에서 사용할 수 있습니다."

#### 2-6. 새 대화 시작 (15초)

새 대화 버튼 클릭

> "새 대화를 시작하면 이전 대화가 초기화되고, 다시 환영 메시지가 표시됩니다."

#### 2-7. 모바일 화면 마무리 (15초)

> "핵심은, 시민이 정확한 메뉴명을 몰라도 자연어로 ChatGPT처럼 대화하면 원하는 페이지로 안내받을 수 있다는 점입니다."

---

### 3. 운영자 대시보드 시연 (2분 30초)

**화면**: http://localhost:8090

#### 3-1. 서비스 정보 카드 (20초)

> "같은 시스템을 운영자 관점에서 보면 이렇습니다. 서비스명, 사이트 ID, Provider, Model, Preset 정보를 한눈에 볼 수 있습니다."

화면 확인:
- 서비스명: AI 홈페이지 파인더
- 사이트 ID: bukgu_gwangju
- Provider / Model / Preset 표시

#### 3-2. 사이트 프로필 카드 (15초)

> "사이트 프로필에는 기관명, 홈페이지 주소, 분류, 중요 키워드, Fallback 전략이 정의되어 있습니다.
> 기관 담당자가 이 정보를 확인하고 수정할 수 있습니다."

#### 3-3. Snapshot 상태 카드 (15초)

> "Snapshot은 홈페이지를 미리 수집해둔 데이터입니다. 로드 성공 여부, 수집 시각, Navigation 링크 수, Source 수를 확인할 수 있습니다.
> 네트워크 없이도 안정적으로 데모할 수 있는 이유입니다."

#### 3-4. LLM 모델 선택 패널 (30초)

> "운영자는 이 패널에서 LLM 모델을 실시간으로 바꿔가며 테스트할 수 있습니다."

화면 확인:
- DeepSeek 기본 (1순위), MiMo 기본 (2순위), Step 기본 (3순위) 라디오 버튼
- 현재 선택된 프리셋 하이라이트

> "기본값은 DeepSeek입니다. MiMo나 Step으로 바꿔서 응답 품질을 비교할 수 있습니다."

#### 3-5. 데모 질문 테스트 (30초)

**[민원서식]** 버튼 클릭

> "운영자가 직접 질문을 테스트할 수 있습니다. 통계를 보면 답변 생성 성공, 출처 건수, Fallback 사용 여부, 경고 수를 확인할 수 있습니다."

화면 확인:
- ✓ 답변 | 출처 N건 | Fallback 여부 | 경고 수
- 출처 테이블 (제목, URL, 유형, 점수)
- 사용된 Provider / Model / Preset 표시

#### 3-6. 운영자 화면 마무리 (15초)

> "운영자 화면은 시민에게 보여주는 화면과 분리되어 있어서, 기관 담당자가 시스템 상태를 모니터링하고 LLM 설정을 조정할 수 있습니다."

---

### 4. 실패 대응 시연 (선택, 30초)

> "API 키가 없거나 네트워크가 불안정한 경우에도 시스템이 안전하게 동작합니다."

확인 포인트:
- API 키 미설정 시: "현재 AI 답변을 생성할 수 없습니다..." 안내 메시지 + 출처 리스트 유지
- Snapshot 모드: 네트워크 없이 사전 수집 데이터로 정상 동작
- Fallback: 검색 결과 없을 시 홈페이지 메뉴에서 출처 자동 생성

---

### 5. 마무리 (30초)

> "지금까지 AI 홈페이지 파인더를 시연했습니다.
> 핵심은 네 가지입니다:
>
> 1. **시민**은 ChatGPT 스타일로 자연어 질문하고 답변과 출처를 받습니다.
> 2. **운영자**는 프로필, Snapshot, LLM 모델을 관리합니다.
> 3. **네트워크 불안정**에도 Snapshot 기반으로 안정적으로 동작합니다.
> 4. **API 키 없이도** StubProvider로 전체 파이프라인을 테스트할 수 있습니다.
>
> 현재는 광주광역시 북구청 1개 기관을 대상으로 PoC를 완성한 상태이며,
> 다른 기관에도 동일한 방식으로 확장할 수 있습니다."

## 질문-답변 예시표

| 질문 | 기대 답변 키워드 | 출처 수 | Fallback |
|------|----------------|---------|----------|
| 민원서식 어디서 받아? | 민원서식, 종합민원 | 2건 | 메뉴 fallback |
| 교육접수는 어디서 해? | 교육접수 | 1건 | 메뉴 fallback |
| 정보공개는 어디서 볼 수 있어? | 정보공개 | 1건 | 메뉴 fallback |
| 고시공고는 어디서 확인해? | 고시공고 | 1건 | 메뉴 fallback |

## 비고

- 모든 데모는 `--snapshot` 모드로 실행하므로 네트워크 불필요
- API 키 불필요 (Snapshot + StubProvider 조합으로 전체 파이프라인 검증 가능)
- 실제 홈페이지 접속 없이 사전 수집 데이터로 동작
- 시연 중 문제가 생기면 서버를 재시작하면 즉시 복구
- 프리셋 미지정 시 DeepSeek 기본(1순위) 자동 적용
- 운영자 대시보드에서 실시간으로 LLM 모델 변경 가능
