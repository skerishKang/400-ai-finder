# Repeated-Question Analytics and Scenario-Cache Promotion Plan

This document defines the planning and strategy for analyzing repeated user questions to safely promote them into the scenario-cache or identify gaps in retrieval.

## 1. Purpose

The objective of repeated-question analytics is to guide optimization:
* **Accelerate common paths**: Promote highly frequent, successfully answered questions to local cache or scenario snapshots (e.g. `review_for_cache` or `review_for_scenario`).
* **Identify gaps**: Pinpoint repeated queries that yield weak or no results, highlighting where indexing, crawler rules, or query rewriting needs enhancement (e.g. `retrieval_gap`).
* **Human-Reviewed Promotion**: Ensure the system does not automatically modify runtime behaviors, code, or databases without human verification.

---

## 2. Inputs from QuestionLogEvent

The input dataset consists of structured `QuestionLogEvent` logs generated by the search-and-answer pipeline. Key fields utilized:
* `normalized_question`: Grouping key to match semantically identical questions.
* `site_id`: To scope candidates per site.
* `answer_status`: Indication of LLM output success ("success", "no_results", "error").
* `source_domains`: Domains used to answer the question, verifying authoritative origin.
* `fallback_used`: Indicates if fallback/navigation sources were used.
* `guard_status`: Status from the source mismatch guard ("pass", "warn", "no_results").
* `warnings`: Capture anomalies or pipeline complaints.

---

## 3. Similar Question Grouping Strategy

* **Normalization**: Clean input text (lowercase, strip particles/special chars).
* **Exact match on normalization**: Group by identical `normalized_question`.
* **Future expansion**: Multi-word semantic match (e.g. Jaccard distance, embedding similarity).

---

## 4. Candidate Promotion Criteria

### A. Cache/Scenario Promotion (`review_for_cache`, `review_for_scenario`)
* **Repeated count** >= threshold (e.g. 3 repeats).
* **High confidence**: Answer status is predominantly `success` (or `pass` status from the mismatch guard).
* **Authoritative**: Source domains represent the official target site.
* **Low fallback count**: Did not rely heavily on generic fallback links.

### B. Retrieval Gap Identification (`retrieval_gap`)
* **Repeated count** >= threshold.
* **Low confidence**: Answer status is predominantly `no_results` or guard status is `no_results` (due to source mismatch).
* **Action**: Review sitemaps, query rewrite expansions, or increase crawl depth.

---

## 5. Human Review Workflow

1. **Analysis Run**: Run offline analyzer on JSONL logs.
2. **Review Candidates**: The analyzer generates a list of `PromotionCandidate` objects.
3. **Manual Verification**: Developers/operators review candidates.
4. **Promotion Action**: Approved cache candidates are promoted to `configs/scenarios/` or similar cache locations.
5. **No Automatic Commits**: No code or data files are modified automatically by the analyzer.

---

## 6. Privacy & Security Rules

* **API Key Redaction**: Never logs or output API keys, secrets, tokens, or credentials.
* **User Anonymity**: No user identifiers (IPs, session IDs, user names) are logged or analyzed.
* **Sanitized Outputs**: Example raw questions in reports must pass through a sanitization function to strip potential private data.

---

## 7. Non-Goals

* Automatic rewrite of configs or scenarios.
* Automatic git commit or pull request creation.
* Real-time production database logging.
* User identity profiling or analytics.

## 9. Dry-run review report

Stage 353 adds a local dry-run report command for sanitized question logs.
The report helps operators review repeated successful questions and retrieval gaps.
It does not create scenarios, snapshots, caches, PRs, or commits.

### Usage

```bash
python scripts/analyze_question_logs.py \
  --input path/to/question-log.jsonl \
  --output path/to/repeated-question-report.md \
  --min-count 3
```

### Report sections

- Summary: input file, total events, candidate counts, timestamp
- Promotion candidates: human-review candidates for cache or scenario promotion
- Retrieval gaps: repeated queries that yield weak or no results
- Notes: explicit dry-run notice and redaction confirmation

### Safety guarantees

- No automatic scenario/snapshot/cache generation
- No PR or commit creation
- No live network, LLM, or fetch calls
- All text fields are sanitized for secret/credential values
- Human review is mandatory before any promotion
