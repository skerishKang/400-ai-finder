# 400-ai-finder 발표 구성안

## 발표 목적

이 발표는 400-ai-finder가 특정 공공기관 사이트를 실시간으로 탐색하고, 실제 LLM을 통해 출처 기반 답변을 생성할 수 있음을 보여주는 데모용 발표 구성안입니다.

예시는 광주광역시 북구청 사이트를 사용합니다.

---

## 전체 발표 흐름

권장 발표 시간: 5~7분
권장 슬라이드 수: 6~7장

---

## Slide 1. 제목

### 제목
400-ai-finder
특정 기관 사이트를 AI와 대화하면서 탐색하는 출처 기반 정보검색 도구

### 핵심 메시지
AI가 일반 지식을 말하는 것이 아니라, 실제 기관 사이트에서 가져온 자료를 바탕으로 답변합니다.

### 발표 멘트
오늘 보여드릴 것은 특정 기관 사이트를 AI가 직접 읽고, 사용자의 질문에 맞춰 출처 기반 답변을 생성하는 데모입니다.
예시는 광주광역시 북구청 사이트입니다.

---

## Slide 2. 문제의식

### 화면 내용
공공기관 사이트 정보는 다음과 같은 한계가 있습니다.

- 공지사항, 고시공고, 민원, 복지, 청년·일자리 정보가 흩어져 있음
- 기관마다 메뉴 구조와 게시판 이름이 다름
- 사용자가 원하는 정보를 찾기 위해 여러 메뉴를 직접 탐색해야 함
- 일반 챗봇 답변은 공식 출처 근거를 확인하기 어려움

### 발표 멘트
공공기관 사이트에는 유용한 정보가 많지만, 일반 사용자가 원하는 정보를 빠르게 찾기는 어렵습니다.
또 AI가 답변하더라도 그 답변이 실제 공식 사이트에 근거했는지 확인할 수 있어야 합니다.

---

## Slide 3. 400-ai-finder의 접근

### 화면 내용
400-ai-finder의 기본 흐름:

1. 사용자가 질문
2. 대상 사이트에서 관련 페이지 fetch
3. LLM이 fetch된 자료를 기반으로 답변 생성
4. 답변과 함께 출처 URL 확인
5. 공식 도메인 grounding 검증

### 발표 멘트
400-ai-finder는 AI가 혼자 답을 추측하는 구조가 아닙니다.
먼저 대상 사이트에서 관련 자료를 가져오고, LLM이 그 자료를 바탕으로 답변합니다.
따라서 답변 아래의 출처 URL을 통해 공식 사이트 근거를 확인할 수 있습니다.

## Slide 4. 북구청 데모 구성

### 화면 내용

| 항목 | 내용 |
|---|---|
| 대상 | 광주광역시 북구청 |
| 도메인 | `bukgu.gwangju.kr` |
| site profile | `bukgu_gwangju` |
| LLM provider | NVIDIA |
| 검증 모델 | `openai/gpt-oss-120b` |
| fetch provider | `requests` |

### 데모 질문
1. 공지사항이나 고시공고는 어디에서 확인할 수 있나요?
2. 민원 신청이나 온라인 민원 안내는 어디에서 확인할 수 있나요?
3. 청년 또는 일자리 관련 정보는 어떻게 찾을 수 있나요?

### 발표 멘트
오늘 데모는 북구청 사이트를 대상으로 진행합니다.
공지사항, 민원, 청년·일자리 질문을 차례로 실행하고, 답변이 어떤 출처에 근거하는지 함께 확인하겠습니다.

---

## Slide 5. 실제 검증 결과

### 화면 내용

| 질문 유형 | 결과 | 비고 |
|---|---|---|
| 공지사항·고시공고 | PASS | 데모 적합 |
| 복지 지원사업 | PASS | 관련 공식 외부 도메인 포함 가능 |
| 민원 신청·온라인 민원 | PASS | 데모 적합 |
| 청년·일자리 | PASS | 데모 적합 |
| 도메인 확인 | WARN | 별도 검색질문보다 답변 출처 확인 방식 권장 |

### 발표 멘트
실제 live fetch와 live LLM을 사용해 검증한 결과, 주요 질문들은 데모에 적합한 답변을 생성했습니다.
도메인 검증은 별도의 검색 질문으로 던지기보다, 각 답변의 출처 URL을 직접 확인하는 방식이 더 안정적입니다.

## Slide 6. 데모 진행 순서

### 화면 내용

1. 북구청 site profile 확인
2. Q1 공지사항·고시공고 질문 실행
3. 답변과 출처 URL 확인
4. Q2 민원 신청·온라인 민원 질문 실행
5. 답변과 출처 URL 확인
6. Q3 청년·일자리 질문 실행
7. 답변과 출처 URL 확인
8. 공식 도메인 grounding 설명

### 발표 멘트
중요한 것은 답변 내용만 보는 것이 아닙니다.
각 답변이 어떤 출처에서 왔는지, 그리고 그 출처가 북구청 공식 도메인인지 함께 확인하는 것입니다.

## Slide 7. 활용 가능성

### 화면 내용

400-ai-finder는 다음 영역으로 확장할 수 있습니다.

- 지자체 홈페이지 민원·복지·공지 안내
- 공공기관 내부 정보 탐색
- 고시공고·지원사업·교육사업 검색
- 시민 대상 북구청 AI 민원 네비게이터
- 특정 기관 사이트 기반 RAG형 챗봇
- 여러 기관 사이트를 대상으로 한 AI 통합 안내 서비스

### 발표 멘트
북구청은 하나의 예시입니다.
같은 구조를 다른 지자체나 공공기관 사이트에 적용하면, 기관별 사이트를 AI와 대화하면서 탐색하는 서비스로 확장할 수 있습니다.

## 예비 질문 대응

### Q. 일반 챗봇과 무엇이 다른가요?

일반 챗봇은 모델이 알고 있는 내용을 바탕으로 답변할 수 있지만, 400-ai-finder는 특정 사이트에서 실제 자료를 가져오고 그 자료에 근거해 답변합니다.
따라서 출처 URL을 통해 공식 도메인 근거를 확인할 수 있습니다.

### Q. 왜 북구청을 예시로 사용했나요?

북구청은 공지사항, 민원, 복지, 청년·일자리처럼 공공기관 사이트에서 흔히 필요한 정보를 포함하고 있어 데모에 적합합니다.

### Q. 검색 결과가 약하게 나오는 경우는 없나요?

있을 수 있습니다.
일부 공공기관 사이트는 자연어 표현과 실제 메뉴명이 다를 수 있습니다.
이 경우 청년, 일자리, 채용, 고용, 비즈광주북구처럼 공식 메뉴에 가까운 대체 검색어를 사용합니다.

### Q. 외부 도메인이 나오면 실패인가요?

항상 실패는 아닙니다.
복지나 고용 정보는 고용복지플러스센터처럼 관련 공식 외부 도메인이 함께 나올 수 있습니다.
이 경우 질문 맥락과 공식성을 함께 확인합니다.

## 발표 마무리 멘트

400-ai-finder의 핵심은 AI가 일반적인 답변을 생성하는 것이 아니라, 특정 공공기관 사이트에서 가져온 자료를 바탕으로 출처 기반 답변을 생성한다는 점입니다.

북구청 예시에서는 공지사항, 민원, 청년·일자리 질문에 대해 실제 출처 기반 답변을 생성하는 흐름을 확인했습니다.

따라서 400-ai-finder는 특정 사이트를 AI와 대화하면서 탐색하는 MVP를 넘어, 제한된 조건에서 외부 시연 가능한 상태입니다.
