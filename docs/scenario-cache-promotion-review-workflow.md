# Scenario/Cache Promotion Review Workflow

## 1. Purpose

This workflow explains how a human reviewer evaluates repeated-question analytics dry-run reports.

The goal is to decide whether repeated questions should become:

- **cache review candidates**
- **scenario review candidates**
- **retrieval improvement items**
- **monitor-only items**

This workflow does **not** automatically create scenarios, snapshots, caches, commits, or pull requests. Human review is mandatory for every decision.

---

## 2. Inputs

A reviewer needs the following inputs to evaluate a dry-run report:

- Sanitized question-log JSONL file
- Dry-run report generated by `scripts/analyze_question_logs.py`
- Source URLs/domains shown in the report
- `answer_status` distribution (PASS / WARN / NO_RESULTS / ERROR)
- `fallback_used` count
- `guard_status` and `guard_reason`
- Query rewrite terms used at retrieval time
- Reviewer's own notes and questions

**Privacy reminder:**

Reviewers should not use raw secrets, API keys, Authorization headers, user identity fields, or unsanitized personal data. The dry-run CLI applies `sanitize_text` redaction to all text fields, but reviewers should still verify that the report does not contain sensitive content before sharing it externally.

---

## 3. Review categories

Each dry-run candidate falls into one of four categories. The categories are mutually exclusive for a single decision, but a candidate may be re-classified later as new evidence emerges.

### 3.1 Cache review candidate

Use when **all** of the following are true:

- The repeated question appears frequently (above the `min-count` threshold)
- The answer is stable across recent events
- The official source domain is stable and well-known
- `answer_status` is mostly PASS
- `fallback_used` is low
- Source mismatch guard does not trigger
- The answer is useful for speed-up but not necessarily needed as a smoke/regression scenario
- The question is not sensitive or personal

### 3.2 Scenario review candidate

Use when:

- The repeated question is important for demo, smoke, regression, or safety coverage
- The official source is stable
- The expected answer behavior can be tested
- The candidate improves product confidence
- A human reviewer can define pass criteria safely
- The scenario does not hardcode volatile facts (e.g. a specific person's name) that would cause the scenario to fail on legitimate site updates

### 3.3 Retrieval gap

Use when:

- The repeated question often returns NO_RESULTS
- `answer_status` is WARN/NO_RESULTS/ERROR-heavy
- `fallback_used` is frequent
- Source mismatch guard triggers
- Source domains are absent, unstable, or off-topic
- Query rewrite terms need improvement
- A relevant official source may exist but is not being found

**Important:** Retrieval gaps are **not** scenario or cache candidates. They require query rewrite, retrieval ranking, source indexing, or guard tuning improvements first.

### 3.4 Monitor-only

Use when:

- The repeated count is low (just above threshold)
- Result quality is mixed
- Official source stability is unclear
- The question may be temporary or event-specific (e.g. "오늘 행사 뭐 있어?")
- The question may contain sensitive or overly personal content
- A future report may provide better signal

## 4. Decision matrix

| Signal | Cache candidate | Scenario candidate | Retrieval gap | Monitor |
|---|---|---|---|---|
| Repeated PASS | Yes | Maybe | No | Maybe |
| Stable official source | Required | Required | Often absent | Unknown |
| High `fallback_used` | No | No | Yes | Maybe |
| Guard mismatch | No | No | Yes | Maybe |
| NO_RESULTS-heavy | No | No | Yes | Maybe |
| Demo/regression value | Maybe | Yes | No | Maybe |
| Sensitive/personal content | No | No | No | Monitor or exclude |
| Volatile/time-sensitive facts | No | No | Maybe | Yes |
| Useful for speed-up | Yes | Maybe | No | Maybe |
| Useful for test coverage | Maybe | Yes | No | No |

**Reading the table:**

- "Yes" / "No" / "Required" / "Often absent" describe the typical signal for that category.
- "Maybe" means the signal alone is not sufficient to decide; combine with other signals.

---

## 5. Human review steps

Follow these steps for each dry-run report:

1. Open the dry-run report Markdown file.
2. Review the **Promotion candidates** section first.
3. Check the listed source domains and verify official source stability.
4. Compare raw and sanitized example questions for clarity and safety.
5. Confirm the question is genuinely repeated and useful (not a one-off typo or noise).
6. Decide the category:
   - Cache review
   - Scenario review
   - Retrieval gap
   - Monitor
7. For cache or scenario candidates, create a **separate** follow-up issue (see templates in section 10).
8. For retrieval gaps, create a query rewrite/retrieval improvement issue.
9. **Do not** commit new scenarios, snapshots, or caches from the dry-run report directly. Promotion is always a separate, human-approved action.

---

## 6. Cache candidate checklist

Before approving a cache candidate, verify:

- [ ] Question repeats above the `min-count` threshold.
- [ ] Answer source is stable and official.
- [ ] Most events show `answer_status = PASS`.
- [ ] `fallback_used` is low.
- [ ] Source mismatch guard does not trigger.
- [ ] No sensitive or personal content is present in the question.
- [ ] Candidate helps reduce repeated live retrieval cost/latency.
- [ ] Cache behavior can expire safely (TTL, staleness policy).
- [ ] Reviewer approved creation of a follow-up issue.

---

## 7. Scenario candidate checklist

Before approving a scenario candidate, verify:

- [ ] Question has demo, smoke, regression, or safety-test value.
- [ ] Expected behavior can be described clearly.
- [ ] Official source domain is stable.
- [ ] Pass criteria can be written without overfitting.
- [ ] Candidate does not hardcode volatile facts unnecessarily.
- [ ] Snapshot/scenario update will be reviewed separately, not auto-committed.
- [ ] No sensitive content is present in the question or expected answer.
- [ ] Reviewer approved creation of a follow-up issue.

---

## 8. Retrieval gap checklist

Before classifying as a retrieval gap, verify:

- [ ] Question repeats but `answer_status` is mostly WARN/NO_RESULTS/ERROR.
- [ ] Query rewrite terms are weak or missing.
- [ ] A relevant official source may exist but is not being found.
- [ ] Source mismatch guard triggers or sources are off-topic.
- [ ] Candidate **should not** be promoted to cache or scenario.
- [ ] Follow-up should target query rewrite, retrieval ranking, source indexing, or guard tuning.

## 9. Examples

All examples below use **synthetic** questions and do not represent real user data.

### Example 1: Cache candidate

```txt
Question group:
- 북구청 민원 신청 어디서 해?
- 온라인 민원 어디서 해?

Status:
- 8 PASS events
- source domains: bukgu.gwangju.kr, eminwon.bukgu.gwangju.kr
- fallback_used: low
- guard_status: pass

Decision: cache review candidate

Reason:
- repeated, stable official source, low mismatch risk, useful for retrieval speed-up
- does not need to be a smoke/regression scenario
```

### Example 2: Scenario candidate

```txt
Question group:
- 고시공고 어디서 봐?
- 북구청 공고 어디서 확인해?

Status:
- 6 PASS events
- source domains: bukgu.gwangju.kr
- stable, well-known menu path

Decision: scenario review candidate

Reason:
- strong demo/smoke/regression value
- expected behavior can be tested with a clear pass criterion
- the answer structure is stable enough to test
```

### Example 3: Retrieval gap

```txt
Question group:
- 구청장이 누구야?
- 북구청장 누구야?

Status:
- repeated NO_RESULTS or source mismatch events
- query rewrite terms may be too narrow
- source domains: none or off-topic

Decision: retrieval gap

Reason:
- improve query rewrite and retrieval ranking first
- do not hardcode a specific person's name as a scenario
- the answer depends on volatile, time-sensitive facts (incumbent officeholder)
```

### Example 4: Monitor-only

```txt
Question group:
- 오늘 행사 뭐 있어?

Status:
- mixed results
- source stability unclear
- likely time-sensitive

Decision: monitor

Reason:
- time-sensitive, may be tied to a specific event window
- future reports may give better signal
- not safe to promote without clearer signal
```

## 10. Follow-up issue templates

Use these templates when creating follow-up issues. They are **not** automation — a human reviewer copies and fills them.

### 10.1 Cache review issue

```md
Title:
[REVIEW] Evaluate cache candidate: <normalized question>

Context:
- Source: repeated-question dry-run report
- Count: <repeat count>
- Site ID: <site id>
- Representative question: <example question>
- Source domains: <list>
- Answer status distribution: PASS <n> / WARN <n> / NO_RESULTS <n> / ERROR <n>

Review checklist:
- [ ] Official source is stable
- [ ] No sensitive content
- [ ] Candidate reduces repeated retrieval cost
- [ ] Cache behavior can expire safely (TTL defined)

Decision:
- [ ] Approve cache implementation follow-up
- [ ] Reject
- [ ] Monitor
```

### 10.2 Scenario review issue

```md
Title:
[REVIEW] Evaluate scenario candidate: <normalized question>

Context:
- Source: repeated-question dry-run report
- Count: <repeat count>
- Site ID: <site id>
- Representative question: <example question>
- Expected source domain: <domain>
- Why this has demo/regression value: <reason>

Review checklist:
- [ ] Expected behavior is stable
- [ ] Pass criteria can be written safely
- [ ] Snapshot/scenario will not hardcode volatile facts
- [ ] No sensitive content

Decision:
- [ ] Approve scenario/snapshot implementation follow-up
- [ ] Reject
- [ ] Monitor
```

### 10.3 Retrieval gap issue

```md
Title:
[RETRIEVAL] Improve retrieval for repeated question: <normalized question>

Context:
- Source: repeated-question dry-run report
- Count: <repeat count>
- Site ID: <site id>
- Representative question: <example question>
- Observed status: WARN <n> / NO_RESULTS <n> / ERROR <n>
- Current query rewrite terms: <list>
- Current source domains: <list>

Suggested direction:
- [ ] Query rewrite expansion
- [ ] Search ranking improvement
- [ ] Source indexing improvement
- [ ] Source mismatch guard tuning
```

---

## 11. Non-goals

This workflow does **not**:

- Automatically create scenarios.
- Automatically create snapshots.
- Automatically create caches.
- Automatically create pull requests.
- Bypass human review.
- Authorize storing sensitive user data.
- Replace the operator question log guide or the dry-run CLI.
