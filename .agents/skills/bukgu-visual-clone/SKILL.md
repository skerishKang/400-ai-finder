# Buk-gu interactive clone and AI browser skill

Use this workflow for Buk-gu clone screens, the chat-to-split experience, and
visible agent actions. The authoritative product direction is
`docs/design/bukgu-ai-agent-product-directive.md`.

## Authority and intent

The project owner has stated that the MVP is produced under an agreement with
Buk-gu leadership and has authorized use of Buk-gu site structure, branding,
public content, and supplied media for the demonstration.

The target is a convincing AI administrative browser, not a static screenshot
viewer. Rich motion, an official-looking semantic clone, visible cursor movement,
clicks, scrolling, search typing, draft writing, and reversible form preparation
are expected product behavior.

## Source rules

1. Preserve hashes and original bytes of ledger source captures.
2. Use approved screenshots as visual evidence, never as a full runtime page.
3. Build controls, layout, lists, tables, forms, and navigation with semantic
   DOM/CSS.
4. Authorized public-site inspection may be used to refresh structure and public
   facts. Record the source URL and capture or retrieval time.
5. Use authorized local assets for identity, portraits, banners, photography,
   video, and artwork.
6. Do not invent official facts. Mark uncertain or stale information clearly.

## Agent interaction rules

- The agent may navigate, search, click, scroll, select, draft, and prefill while
  keeping each action visible to the resident.
- Prefer an on-screen AI cursor and human-readable action narration.
- Type search terms and drafts character by character when the action is part of
  the presentation.
- Let the resident interrupt, cancel, edit, or take over.
- Require explicit confirmation immediately before final submission, publishing,
  payment, signature, identity verification, personal-file upload, or destructive
  record changes.
- Do not persist sensitive personal data in analytics, browser storage, fixtures,
  screenshots, or model logs.

## Visual workflow

1. Confirm the relevant reference viewport and current product route.
2. Reconstruct hierarchy and interaction states before decorative refinement.
3. Compare the local render with the reference at the matched viewport.
4. Exercise hover, click, keyboard, search, typing, route changes, and reduced
   motion in a local browser.
5. Report remaining visual differences and distinguish them from functional
   failures.

## Repository safety

- Preserve unrelated user changes in a dirty worktree.
- Do not reset, clean, stash, rebase, force-push, or alter `main` without an
  explicit owner request.
- Run focused tests, syntax checks, `git diff --check`, and a browser smoke pass
  for edited interaction code.
